# MyOS development

## M0 status

M0 establishes the reproducible build boundary. The repository now contains the canonical
agent protobuf API, an initial Rust daemon process, a local pacman package pipeline, and an
ArchISO profile overlay that produces a branded plain-TTY live image.

The first successful local build produced `out/myos-26.07.11-x86_64.iso` (1.6 GB). Its El
Torito catalog contains bootable BIOS and UEFI entries and the image has an isohybrid MBR/GPT.

The ArchISO overlay is applied to the `releng` profile shipped by the container's installed
`archiso` package. This intentionally keeps boot assets synchronized with rolling Arch while
the files under `os/archiso/` remain the MyOS-owned configuration.

## Windows setup

1. Start Docker Desktop with WSL2 integration, or install Podman.
2. In Ubuntu/WSL2, install the small host prerequisites with
   `sudo apt update && sudo apt install -y make`.
3. Run commands from WSL2 so GNU Make and the shell scripts are available.
4. From the repository root, run `make doctor`, then `make test`.
5. Build the image with `make iso`. The result is written under `out/`.

The ISO build requires a privileged Linux container because `mkarchiso` uses mounts and loop
devices. Docker Desktop must be using Linux containers. Downloaded Arch packages are retained
in the Docker volume `myos-pacman-cache`, while temporary rootfs work stays on the container's
native Linux filesystem for acceptable WSL2 performance.

## Boot acceptance test

Add `myos.autotest=1` to an ISO boot entry. A successful M0 boot writes
`MYOS-M0-BOOT-OK` to the first serial port. This is the signal CI will consume when a
nested-KVM runner is added.

## M1 status

M1 replaces the plain TTY with a graphical session and a working agent:

- **Session**: `greetd` (tty1) → `cage` (Wayland kiosk) → `myos-shell` (Flutter, GTK embedder).
  Config in `os/archiso/airootfs/etc/greetd/config.toml`; enabled by symlinks created in
  `scripts/prepare-archiso-profile.sh` (default target → graphical, greetd wanted). The live
  session runs as root, like stock archiso; the M3 installer will switch it to a real user.
- **Daemon**: `myosd` now implements the full `Agent` gRPC service over
  `/run/myos/agent.sock` — streaming chat, provider connect (API key, validated via a free
  models-list call), provider/model selection, device profile. Provider adapters speak raw
  HTTP: Anthropic Messages API (SSE) and OpenAI chat completions (SSE). Keys are stored in
  `/var/lib/myos/providers.toml` (0600) until the M2 vault lands.
- **Shell**: `shell/` is a Flutter app — bottom chat bar, provider connect sheet (⚡), model
  picker chip, streaming conversation view, and a real terminal (`xterm` + `flutter_pty`) on
  **Ctrl+Shift+T**. Dart gRPC bindings are generated at build time by `shell/tool/gen_protos.sh`;
  the `linux/` runner is generated by `flutter create` on first build. With no provider
  connected the daemon echoes; after connecting a key, chat is live end-to-end.
- **Build**: the toolchain container now bakes in the Flutter SDK and `protoc-gen-dart`.
  `scripts/build-packages.sh` builds the release daemon binary and the Flutter bundle, then
  packages them as `myosd` and `myos-shell` pacman packages consumed by the ISO.

### M1 boot acceptance test

With `myos.autotest=1`, a successful boot writes `MYOS-M0-BOOT-OK` and then
`MYOS-M1-BOOT-OK` (agent socket present + greetd active) to the first serial port.
