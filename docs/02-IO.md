# MyOS — I/O Guide (input, display, audio, network, storage, device mesh)

> **Goal:** every path bytes take in and out of a MyOS device, which Linux subsystem owns it,
> which MyOS component consumes it, and the exact stack choices. Covers keyboard/mouse/touch,
> TV remotes, display, speakers/mic (incl. wake word capture), networking, Bluetooth, storage,
> and the full phone↔PC mesh protocol (`myos-linkd`).

---

## 1. Input stack

```
hardware → kernel drivers → /dev/input/event* (evdev)
        → libinput (inside the compositor, cage/wlroots)
        → Wayland protocol events → Flutter GTK embedder → shell widgets
```

Rules:

- **Never read evdev directly in the shell.** The compositor owns input. Exceptions: `myos-voiced` (audio, not evdev) and `myos-linkd`'s *injector* which uses `/dev/uinput` to synthesize input (§8.4).
- Keyboard layout/config via xkbcommon; set from onboarding choice (`/etc/myos/input.toml` → cage env `XKB_DEFAULT_LAYOUT`).

### 1.1 Global shortcuts (the terminal hotkey)

`Ctrl+Shift+T` / `Cmd+Shift+T` (Super acts as Cmd on Mac keyboards) must work *system-wide*, even if a chat view has focus:

- v1 (single fullscreen Flutter surface): handle in the shell's top-level `Focus`/`Shortcuts` widget — it sees all keys since it owns the whole screen. Map both `LogicalKeySet(ctrl, shift, keyT)` and `(meta, shift, keyT)`.
- Also register a compositor-level binding as a safety net so a frozen shell can still get a terminal: cage has no bindings, so run a tiny sidecar (`myos-hotkeyd`, reads evdev with CAP `input` group) that, on the chord, asks systemd to spawn `foot`/recovery terminal as an overlay — v2 nicety, skip for M1.
- Reserve now, implement later: `Super+Space` = focus chat bar, `F1` long-press on TV remote = voice.

### 1.2 Touch

- Touchscreens: evdev ABS_MT → libinput → Wayland touch events → Flutter handles natively (Flutter is touch-first; nothing special).
- Gestures (phone form factor): implement in Flutter (`GestureDetector`), not libinput gestures — keeps one codebase behavior.
- Calibration for weird panels: `libinput` quirks files in `/etc/libinput/local-overrides.quirks`.

### 1.3 TV remotes

Three transport types, all normalized to key events before the shell sees them:

| Remote type | Path | Notes |
|---|---|---|
| **HDMI-CEC** (TV's own remote) | kernel `cec` framework → `libcec`/`cec-ctl` → `myos-cecd` translates to uinput key events (DPAD_*, SELECT, BACK) | Must also *send* CEC: "TV on" / input switching when MyOS box wakes |
| **IR receiver** | kernel `rc-core` → keymap via `ir-keytable` (`/etc/rc_keymaps/`) → arrives as normal evdev keyboard | ship keymaps for common MCE remotes |
| **Bluetooth remote / gamepad** | BlueZ HID → evdev | pairing UI in settings |

Shell TV layout consumes only: DPAD up/down/left/right, SELECT, BACK, HOME, MIC. Everything must be reachable with those six + voice.

### 1.4 Virtual/injected input (mesh)

`myos-linkd` creates a uinput device (`/dev/uinput`, udev rule grants group `myos-link`) named `MyOS Remote Input`. Phone-side touches map to either:
- **touchpad mode**: relative motion + taps (phone = giant trackpad for PC), or
- **absolute mode**: direct mapping when mirroring/dual-screen (touch where you see).

## 2. Display stack

```
GPU → DRM/KMS (kernel modesetting) → wlroots (cage) → Wayland → Flutter GTK embedder (EGL/GLES)
```

- **cage** runs the shell as its only client, fullscreen, no decorations: `cage -d -- myos-shell`.
- GPU drivers to ship: `mesa` (covers intel/amd/virtio-gpu/**vmwgfx** via gallium — VMware 3D accel works with `open-vm-tools` + SVGA II adapter), `nvidia` deferred (nouveau in image; proprietary later via driver add-on).
- Multi-monitor (desktop v1): cage spans/duplicates; proper per-output layout needs replacing cage with a thin custom wlroots compositor (`myos-comp`) — planned v2, keep the shell compositor-agnostic (pure Wayland client).
- **Virtual outputs for dual-screen** (§8.6): requires wlroots' headless/virtual output — another reason `myos-comp` eventually replaces cage. For M6, alternative that works under cage: stream a *rendered Flutter route* (a second logical desktop drawn by the shell itself) instead of a real compositor output. Cheaper, good enough for chat-era UX.
- Fractional scaling: Flutter handles via `wp_fractional_scale_v1`; TV = force 2.0 scale.
- Screen capture for streaming: `wlr-screencopy` / PipeWire screencast portal (`xdg-desktop-portal-wlr`) → feeds the encoder (§8.6).

## 3. Audio stack (out) and microphone (in)

**PipeWire everywhere** (+ WirePlumber session manager). Kill pulseaudio/JACK questions permanently — PipeWire emulates both.

- Shell volume control: `wpctl` equivalents via PipeWire native API (Rust `pipewire` crate in `myosd` tool `settings.volume`).
- Per-device profiles: laptop speakers, HDMI audio on TV (default sink = HDMI when form factor = TV), BT headsets (BlueZ + PipeWire's BAP/A2DP).
- **TTS playback**: `myos-voiced` opens a dedicated PipeWire stream with `media.role=voice-assistant`; WirePlumber policy ducks other audio 60% while the agent speaks.

### 3.1 Mic + wake word capture path (hot path, always on)

```
mic → PipeWire capture stream (16 kHz mono, myos-voiced)
    → ring buffer (2 s)
    → VAD (silero-vad, ~1 MB ONNX)                    [always running, ~0% CPU when silent]
    → wake-word engine on speech frames               [openWakeWord ONNX model]
    → on trigger: stream buffered audio + live audio to STT (whisper.cpp local, or provider STT)
    → transcript → myosd as a user message (marked source=voice)
```

- **Wake word = custom agent name.** openWakeWord supports training a custom model from synthetic TTS samples: at onboarding, `myos-voiced` generates ~2000 TTS variations of "hey `<AgentName>`" (Piper voices at varied speeds/pitches) and fine-tunes the small classifier **on device** (takes ~1–2 min CPU) or falls back to phonetic matching via whisper-tiny streaming when training is not feasible on weak hardware. Store model at `/var/lib/myos/wakeword/<name>.onnx`. This is the hardest part of voice — prototype it early (M4 spike), fallback = fixed wake word "Hey MyOS" if custom training proves unreliable.
- Privacy invariant: **no audio leaves the device until after wake-word trigger**, and a shell indicator (mic glyph) is mandatory whenever a capture stream is open. Hardware mute keys → `settings.mic_mute` respected at PipeWire level.
- Echo cancellation: PipeWire `filter-chain` AEC module so voice works while music plays (TV case especially).

### 3.2 TTS (agent voice)

**Piper** (fast, local, many voices, ONNX). Voice chosen at onboarding. Streamed sentence-by-sentence as the provider streams text — do not wait for the full reply. Provider-native TTS (e.g. cloud voices) optional later behind the same `voice.speak` interface.

## 4. Networking

- **NetworkManager** (+ `iwd` as wifi backend — faster scans/connects than wpa_supplicant).
- Shell talks to NM via D-Bus (Flutter: through `myosd` tools `net.*`, never directly).
- Onboarding wifi screen = `net.scan` + `net.connect_wifi(ssid, psk)` tools reused by the agent later ("connect to hotel wifi").
- Captive portal detection: NM's connectivity check → shell shows an embedded webview? **No webview in v1** — show "open portal on your phone" + QR of gateway URL; revisit later.
- mDNS: `systemd-resolved` with mDNS enabled (linkd depends on it, §8.1). Firewall: `nftables` default-deny inbound except linkd's QUIC port (49152) and mDNS — ruleset in `os/rootfs-overlay/etc/nftables.conf`.
- Time: `systemd-timesyncd`. Hostname: set from onboarding device name (`myos-livingroom-tv`).

---

## 5. Bluetooth

- **BlueZ**, controlled via D-Bus from `myosd` (`bt.*` tools: scan, pair, connect, forget).
- Priority profiles: HID (keyboards/remotes/gamepads), A2DP/BAP (audio), later LE peripherals.
- Pairing UX runs in shell settings; agent can drive it ("pair my headphones") with a MED-risk confirm.

---

## 6. Storage & removable media

- Internal layout: doc 01 §2. Mounts via systemd + gpt-auto; `/etc/fstab` only for /var and /home.
- USB drives/SD: **udisks2**; auto-mount to `/run/media/<user>/<label>` when the session is active; shell shows a toast ("USB drive connected — ask me to copy something?").
- Agent access: `fs.*` tools include mounted removable paths but *writing to them is MED risk* and eject (`disk.eject`) is exposed as a tool.
- MTP/PTP (cameras, Android phones): defer; the MyOS mesh replaces most of it.

---

## 7. Sensors & misc I/O

- Backlight: `/sys/class/backlight` via `systemd-logind`'s `SetBrightness` D-Bus (no udev rule hacks) → `settings.brightness`.
- Battery/AC: `upower` → shell status + agent tool `system.battery`.
- Ambient light / accelerometer (phone/tablet later): `iio-sensor-proxy` → auto-rotation, auto-brightness.
- Webcam: v4l2; only used when user/agent explicitly requests (same indicator rule as mic).
- Printing: CUPS, defer to post-v1 (agent tool `print.file` stub documented, unimplemented).

## 8. Device mesh — `myos-linkd` protocol (phone ↔ PC ↔ TV)

The flagship I/O feature. One daemon, one QUIC connection per peer, multiplexed channels.

### 8.1 Discovery

- mDNS/DNS-SD via systemd-resolved, service `_myos-link._udp.local`, port 49152.
- TXT records: `v=1 name=<device name> ff=<phone|desktop|tv> fp=<sha256 of device cert> proto=quic`.
- Each device generates a self-signed Ed25519 device certificate at first boot (`/var/lib/myos/link/device.{key,crt}`); the **fingerprint is the device identity**.
- Off-LAN (future, v2+): optional relay/rendezvous server, same mTLS inside; document but don't build.

### 8.2 Pairing (trust establishment)

```
Initiator (phone)                    Responder (PC)
  discovers PC via mDNS
  "pair" → PC shows QR: {fp, ip, port, spake2_session_id}
  phone scans QR  ──────────────────►
  SPAKE2 handshake using QR nonce as password  (PIN fallback: PC shows 6 digits, user types on phone)
  both sides verify, then exchange device certs inside the SPAKE2-secured channel
  both persist: /var/lib/myos/link/peers/<fp>.toml  {name, cert, capabilities, granted: [...]}
```

- After pairing, every connection is **mTLS 1.3 over QUIC** (quinn crate), both sides pin the stored peer cert. No CA, no cloud.
- Revocation = delete peer file (Settings → Devices → Forget). Capabilities granted per-peer: `clipboard`, `files`, `notifications`, `input`, `screen` — each toggleable; `input` and `screen` default **off** until first use prompts on the *controlled* device.

### 8.3 Channels (QUIC streams)

| Channel | Stream type | Payload |
|---|---|---|
| `ctrl` | bidi, reliable | JSON-RPC 2.0: capability negotiation, session mgmt, agent `link.*` calls |
| `clip` | bidi, reliable | clipboard sync (text/image), debounced, size-capped 5 MB |
| `file` | uni per transfer | length-prefixed chunks + BLAKE3 whole-file hash; resume via offset |
| `input` | bidi, reliable, low-latency | binary-packed input events (§8.4) |
| `notif` | uni | mirrored notifications (phone→PC etc.) |
| `av` | uni, **unreliable datagrams** | RTP-style H.264/HEVC video + Opus audio (§8.6) |

### 8.4 Remote input ("control PC from mobile")

- Wire format: 16-byte packed events `{u64 timestamp_us, u16 type, u16 code, i32 value}` — literally evdev's struct; zero translation cost.
- Receiver injects via its uinput device (§1.4). Modes negotiated on `ctrl`: `touchpad` (REL_X/REL_Y + BTN), `absolute` (ABS_X/ABS_Y scaled to receiver resolution), `keyboard` (KEY_*).
- Phone UI (`shell/lib/link/remote_control.dart`): trackpad surface + keyboard button + scroll edge + a quick-actions row (media keys, Esc, Ctrl+Shift+T!).
- **Consent rule:** first input event of a session requires the controlled device to show "Allow `<phone>` to control this device? Allow / Allow always / Deny." A persistent status-bar chip shows while any remote-input session is live; tapping it kills the session. Local physical Escape-hammer (press Esc 3× fast) also kills it.

### 8.5 Clipboard, files, notifications

- Clipboard: watch Wayland clipboard (wlr-data-control protocol) → push on change to peers with `clip` granted. Loop-prevention via content hash.
- Files: agent tool `link.send(path, device)` or drag onto the Devices panel. Receive to `~/Inbox/<peer>/` + notification.
- Notifications: MyOS's own notification service (`myosd` `notify.*`) forwards to peers per policy (phone→PC on, PC→phone off by default).

### 8.6 Screen streaming / dual screen

- **Capture:** PipeWire screencast (portal) node → GStreamer/direct pipeline.
- **Encode:** VA-API (`h264_vaapi`) where available; x264 `ultrafast/zerolatency` software fallback; 60 fps target, dynamic bitrate 5–25 Mbps LAN, keyframe on demand (receiver requests on loss).
- **Transport:** QUIC datagrams, custom minimal RTP-ish header `{u32 frame, u16 chunk, u16 chunks_total, flags}`; receiver-driven: NACK → keyframe request. Target glass-to-glass **<60 ms** LAN.
- **Decode/present:** phone side decodes via V4L2/VA-API into a Flutter external texture (`Texture` widget) — fullscreen route with touch mapped to `absolute` input channel pointed back at the PC.
- **Modes:**
  - *Mirror* — stream the primary output.
  - *Extend (dual screen)* — v1 trick: shell renders a second logical workspace offscreen (Flutter `SceneBuilder` to FBO) and streams that; windows/routes can be "moved" to it. True compositor virtual output comes with `myos-comp` (v2). PC-extends-to-phone and phone-extends-to-PC are the same code path with roles swapped.
- Audio: optional Opus stream on `av` (TV use case: phone as remote *speaker* is off by default).

### 8.7 Agent integration

`link.*` tools exposed by `myosd` (backed by linkd's ctrl channel): `link.devices()`, `link.send(file, device)`, `link.control(device)`, `link.screen(device, mode)`, `link.notify(device, msg)`. So "put my laptop screen on the TV" is one voice command → `link.screen(tv, mirror)`.

## 9. Acceptance checklist (I/O done when…)

- [ ] Keyboard, mouse, touchpad, touchscreen all drive the shell in QEMU, VMware, and one real laptop.
- [ ] `Ctrl+Shift+T` opens terminal from every screen of the shell.
- [ ] Volume/brightness/mic-mute keys work and show shell OSD.
- [ ] Wake word triggers with custom name at <1 false accept / hour ambient, <500 ms trigger latency; mic indicator never lies.
- [ ] Voice round-trip (wake → ask time → spoken answer) < 3 s with local STT/TTS.
- [ ] Two VMs pair via PIN, sync clipboard both ways, transfer a 1 GB file with hash verify + resume.
- [ ] Phone-mode VM controls desktop VM cursor at <30 ms input latency LAN; consent chip visible; Esc×3 kills.
- [ ] Screen mirror runs 1080p60 <60 ms on LAN with VA-API; falls back to software encode.
- [ ] All mesh traffic verified mTLS (packet capture shows nothing readable); unpaired device gets connection refused.
